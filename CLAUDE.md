# CLAUDE.md

行为准则，旨在减少常见的 LLM 编码错误。必要时可与项目特定指令合并。

**权衡取舍**：本指南倾向于谨慎而非速度。对琐碎任务可自行判断。

**注意**：所有回答使用中文回复。

## 1. 编码前先思考

**不要假设。不要隐藏困惑。要展示权衡。**

在实现之前：
- 明确说明你的假设。如有不确定，请提问。
- 若存在多种解释，全部列出——不要自行挑选。
- 若有更简洁的方案，指出并建议采用。必要时提出异议。
- 若有不清楚的地方，停下来说明模糊之处并询问。

## 2. 简单优先

**只写能解决问题的最小代码。不要做投机取巧的功能。**

- 不要实现超出需求之外的功能。
- 不要为单次使用编写抽象层。
- 不要加入“灵活性”或“可配置性”，除非有人明确要求。
- 不要为不可能出现的场景写错误处理。
- 如果可以用 50 行代码完成，却写了 200 行，请重构。

思考：**资深工程师会不会觉得这段代码过于复杂？** 若答案是“会”，就要简化。

## 3. 外科手术式修改

**只触及必须更改的部分，只清理自己产生的垃圾。**

编辑已有代码时：
- 不要“顺手”改动相邻的代码、注释或格式。
- 不要重构并非错误的代码。
- 保持与现有风格一致，即使你本人的风格不同。
- 若看到与本任务无关的死代码，提出来——不要自行删除。

如果你的修改导致了孤儿代码（未使用的导入/变量/函数），应当删除。除非用户明确要求，否则不要删除历史遗留下来的死代码。

**验证原则**：每一行修改都必须直接对应用户的请求。

## 4. 目标驱动执行

**定义成功标准，循环验证直至达成。**

将任务转化为可验证的目标：
- “添加校验” → “为非法输入写测试，然后让它们通过”
- “修复 bug” → “编写能复现该 bug 的测试，然后使其通过”
- “重构 X” → “确保重构前后测试全部通过”

对于多步骤任务，简要列出计划：
```
1. [步骤] → 验证: [检查点]
2. [步骤] → 验证: [检查点]
3. [步骤] → 验证: [检查点]
```

明确的成功标准可以让你自行循环迭代。若只有“让它可用”这类宽泛目标，则需要持续沟通确认。

---

**这些准则的效果体现在：**
- 提交的 diff 中不出现不必要的改动。
- 减少因为代码过度重构而产生的返工。
- 在实施前先提出澄清问题，而不是在出现错误后才纠正。
